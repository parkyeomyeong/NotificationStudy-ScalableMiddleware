# Test 2 - 멀티 스레드 컨슈머

## 목적

mock 서버 부하 테스트로 파악한 최적 동시 요청 수(VU 200)를 기준으로 컨슈머 스레드 풀 크기를 결정하고,
Test 1과 동일한 15,000건 부하에서 처리량이 얼마나 개선되는지 수치로 증명

## 스레드 풀 크기 결정 근거

mock 서버 최대 동시 처리: **200**

ERROR 발생 시 최대 3회 재시도 → 실패 큐 총 유입량:
```
N/3 + N/9 + N/27 = 13N/27 ≈ 0.48N

Main : Failed 비율 = N : 13N/27 = 27 : 13
```

| 큐 | 스레드 수 | 근거 |
|----|---------|------|
| Main | **120** | 200 × 27/40 = 135|
| Reserved | **20** | 여유분 |
| Failed | **60** | 200 × 13/40 = 65|
| **합산** | **200** | mock 서버 한도 |

## 이론값 대비 실측값 분석

이론상 120 스레드면 Test 1 대비 **120배** 향상이 예상되지만 실측은 **36.7배**에 그쳤다.

**원인:**

**① 실패 큐 스케줄러 배치 병목**
실패 큐 스케줄러가 5초마다 100건씩만 DB에서 가져옴.
12,000건 이상의 FAILED 적체 시 스케줄러 처리량 상한 = 100/5초 = **20건/s**.
60개 스레드가 있어도 먹이를 20건/s 속도로밖에 못 받음.

**② mock 서버 과부하로 ERROR 비율 증가**
120 스레드가 4초씩 커넥션을 물고 동시 요청하는 부하는,
k6 VU 테스트와 성격이 다름. 서버가 33% 이상의 ERROR를 반환하면서
재시도 사이클이 예상보다 많이 발생.
