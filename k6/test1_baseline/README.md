# Test 1 - Baseline

## 이 시스템의 성능 지표

이 미들웨어는 **비동기 2계층 구조**다.

```
[계층 1] API 수용 레이어          [계층 2] 컨슈머 처리 레이어
  /notifications/regist   →  DB  →  스케줄러  →  Queue  →  Consumer  →  Mock 서버
  "등록 받음" 즉시 200 반환          "실제 발송 처리"
```

k6로 측정하는 API TPS는 **계층 1의 속도(DB write 속도)**일 뿐
이 미들웨어의 실제 성능은 **계층 2**에서 결정

### 성능 지표

| 지표 | 정의 | 측정 방법 |
|------|------|----------|
| **Throughput** | 초당 몇 건의 알림을 실제 발송하는가 | DB 단위시간당 처리 건수 |
| **E2E Latency** | 등록 후 실제 발송까지 걸리는 시간 | `lastTriedAt - createdAt` |

## 테스트 환경

| 항목 | 값 |
|------|---|
| 채널 | TEAMS |
| 컨슈머 스레드 | **1개 (단일)** |
| RestTemplate 타임아웃 | **없음** |
| Mock 서버 평균 응답시간 | 4초 |
| Mock 서버 결과 분포 | SUCCESS 33% / FAILURE 33% / ERROR 33% |
| 부하 패턴 | 100 VU, 고정 **15,000건** (shared-iterations) |

> Test 2/3과 동일한 15,000건 고정 부하. 컨슈머 처리량 비교를 위해 모든 테스트 동일 조건 사용.

## 계층 2 측정 결과 - 컨슈머 처리 성능 (DB)

k6 종료 후 **300초(5분)** 경과 시점에서 H2 Console 측정

### 처리 현황

```sql
SELECT status, COUNT(*) AS count
FROM notification_request
GROUP BY status ORDER BY count DESC;
```

| STATUS | COUNT |
|--------|-------|
| PENDING / QUEUED | **14,919건** |
| SUCCESS / FAILED / PERMANENT_FAILED | **81건** |

### Throughput

```
처리 건수 : 81건
경과 시간 : 300초 (5분)
Throughput : 81 / 300 ≈ 0.27건/s
```

### E2E Latency (잔여 적체 기준)

```
총 등록 건수  : 15,000건
5분 후 처리량 : 81건 (0.54%)
잔여 적체     : 14,919건
처리 속도     : 0.27건/s
잔여 처리 예상 시간 : 14,919 / 0.27 ≈ 55,256초 ≈ 15.3시간
```

---

## 결론

> 단일 스레드 컨슈머는 실사용이 불가능한 수준이다.
>
> Test 2 기준으로 5분이면 처리 완료되는 15,000건을,
> 단일 스레드는 5분 동안 81건(0.54%)밖에 처리하지 못한다.
> 나머지 14,919건을 소화하려면 약 **15.3시간**이 걸린다.
>
> 처리 속도가 0.25건/s라는 수치 하나로 테스트를 끝까지 볼 필요조차 없다.
> 이미 이 아키텍처는 실사용 불가.
