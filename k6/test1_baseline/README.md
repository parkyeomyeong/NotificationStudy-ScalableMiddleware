# Test 1 - Baseline

## 이 시스템의 성능 지표

이 미들웨어는 **비동기 2계층 구조**다.

```
[계층 1] API 수용 레이어          [계층 2] 컨슈머 처리 레이어
  /notifications/regist   →  DB  →  스케줄러  →  Queue  →  Consumer  →  Mock 서버
  "등록 받음" 즉시 200 반환          "실제 발송 처리"
```

k6로 측정하는 API TPS는 **계층 1의 속도(DB write 속도)**일 뿐
이 미들웨어의 실제 성능은 **계층 2**에서 결정

### 성능 지표

| 지표 | 정의 | 측정 방법 |
|------|------|----------|
| **Throughput** | 초당 몇 건의 알림을 실제 발송하는가 | DB 단위시간당 처리 건수 |
| **E2E Latency** | 등록 후 실제 발송까지 걸리는 시간 | `lastTriedAt - createdAt` |

---

## 테스트 환경

| 항목 | 값 |
|------|---|
| 채널 | TEAMS |
| 컨슈머 스레드 | **1개 (단일)** |
| RestTemplate 타임아웃 | **없음** |
| Mock 서버 평균 응답시간 | 4초 |
| Mock 서버 결과 분포 | SUCCESS 33% / FAILURE 33% / ERROR 33% |
| 부하 패턴 | 1 → 200 VU 점진 증가 (총 3분) |

## 계층 1 측정 결과 - API 수용 속도 (k6)

```
총 요청 수   : 162,926건
API TPS      : 904.88 req/s
에러율       : 0%
p50 latency  : 1.24ms
p95 latency  : 4.94ms
```

> 이 수치는 "얼마나 빨리 DB에 저장하는가"이지 "알림이 얼마나 빨리 발송되는가"가 아님

## 계층 2 측정 결과 - 컨슈머 처리 성능 (DB)

k6 종료 후 **1,398초(약 23분)** 경과 시점에서 H2 Console 측정

### 처리 현황

```sql
SELECT status, COUNT(*) AS count
FROM notification_request
GROUP BY status ORDER BY count DESC;
```

| STATUS | COUNT |
|--------|-------|
| PENDING / QUEUED | **162,552건** |
| SUCCESS / FAILED / PERMANENT_FAILED | **374건** |

### Throughput

```
처리 건수 : 374건
경과 시간 : 1,398초
Throughput : 374 / 1,398 ≈ 0.27건/s
```

### E2E Latency (잔여 적체 기준)

```
현재 적체 : 162,552건
처리 속도 : 0.27건/s
잔여 처리 예상 시간 : 162,552 / 0.27 ≈ 601,000초 ≈ 7일
```

---

## 결론

> 단일 스레드 컨슈머는 실사용이 불가능한 수준이다.
>
> API는 초당 904건 이상을 받을 수 있지만 컨슈머는 약 0.25건/s 처리가 한계.
> 23분이 지난 시점에도 등록된 알림의 **99.77%가 미발송 상태**로 적체된다.
> 이 상태에서 마지막으로 등록된 알림이 실제로 발송되려면 약 **7일**이 걸린다.
>
> 처리 속도가 약 0.25건/s라는 수치가 나오는 순간, 테스트를 끝까지 지켜볼 필요조차 없다.
> 이미 이 아키텍처는 실사용 불가.
>
> API 수용 속도가 아무리 빨라도 컨슈머 처리 속도가 따라가지 못하면
> 시스템 전체가 의미없다고 판단.
